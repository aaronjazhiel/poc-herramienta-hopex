Alta de Sistema Aplicativo vía API — HOPEX GraphQL

Automatización del registro de aplicaciones en la plataforma de arquitectura empresarial

Este flujo demuestra cómo registrar un Sistema Aplicativo completo en HOPEX utilizando exclusivamente las APIs GraphQL expuestas en Pre-Producción. El proceso cubre desde la validación de existencia hasta la vinculación de todos los componentes y asignación de responsables, eliminando la necesidad de captura manual en la interfaz web de HOPEX.

📌 Ambiente: Pre-Producción (pprod-gnp.hopexcloud.com)
🔑 Autenticación: Header x-api-key
📡 Método: POST con body JSON (GraphQL queries/mutations)
📦 Módulos: ITPM (Portfolio) + ITARC (Arquitectura)

Lo que NO se puede hacer con las APIs

Limitaciones identificadas durante la POC

  • ❌ Eliminar objetos
    No existe mutation de delete en la API GraphQL. Solo se puede desde la UI de administración de HOPEX.
  • ❌ Saber quién borró un objeto
    No hay endpoint de auditoría de eliminaciones. El Repository Log solo es accesible desde la UI admin.
  • ❌ Gestionar permisos / roles de usuarios
    La administración de usuarios y permisos es exclusiva de la consola de administración.
  • ❌ Acceder al Repository Log completo
    El historial completo de cambios (quién hizo qué y cuándo) no está expuesto vía API.
  • ❌ Generar diagramas o vistas gráficas
    Los diagramas de arquitectura solo se generan desde la interfaz web de HOPEX.
  • ❌ Ejecutar workflows de aprobación
    Los flujos de aprobación (statusReview → Approved) requieren intervención manual en la UI.
  • ❌ Crear/modificar el esquema GraphQL
    El schema está definido por MEGA (proveedor de HOPEX) y no es extensible vía API.
  • ❌ Exportar reportes o dashboards
    Los reportes predefinidos solo se exportan desde la UI web.
  • ❌ Mover objetos entre ambientes (pprod → prod)
    La promoción entre ambientes es un proceso administrado por el equipo de HOPEX/MEGA.
